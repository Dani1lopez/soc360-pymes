# Dependency Management Specification

## Purpose

Define the uv project-mode packaging contract and pip rollback path.

## Requirements

### Requirement: Source of Truth Files

The dependency contract SHALL be `pyproject.toml`, `.python-version`, and `uv.lock`.

#### Scenario: Files are tracked

- GIVEN a fresh clone
- WHEN `git ls-files` runs
- THEN `pyproject.toml`, `.python-version`, and `uv.lock` SHALL be listed

### Requirement: Reproducible Resolution

The system SHALL resolve dependencies via `uv sync`. The lockfile SHALL be generated by uv and treated as generated output.

#### Scenario: Frozen sync reproduces the tree

- GIVEN committed `pyproject.toml` and `uv.lock`
- WHEN `uv sync --frozen --extra dev` runs
- THEN `.venv` SHALL contain the exact pinned tree

### Requirement: Runtime and Development Separation

The system SHALL declare runtime dependencies under `[project.dependencies]` and development tooling under `[project.optional-dependencies.dev]`.

#### Scenario: Sync targets the correct set

- GIVEN runtime and dev optional dependencies exist
- WHEN `uv sync --frozen` and `uv sync --frozen --extra dev` each run
- THEN only runtime deps install first, and dev tools also install second

### Requirement: Extras Preservation

Extras declared in `pyproject.toml` SHALL be preserved and not flattened.

#### Scenario: Extras resolve

- GIVEN `pyproject.toml` declares `uvicorn[standard]`, `sqlalchemy[asyncio]`, `redis[hiredis]`, `celery[redis]`, `python-jose[cryptography]`, and `passlib[bcrypt]`
- WHEN `uv sync --frozen` resolves
- THEN extra-provided packages (e.g., `httptools`, `cryptography`, `bcrypt`, `hiredis`) SHALL be in `.venv`

### Requirement: Command Surface

The system SHALL expose project commands through `uv run` for `pytest`, `ruff`, `mypy`, `alembic`, and `celery`.

#### Scenario: uv run executes commands

- GIVEN `uv sync --frozen --extra dev` succeeded
- WHEN the developer runs `uv run pytest`, `uv run ruff check .`, `uv run mypy .`, `uv run alembic upgrade head`, or `uv run celery --help`
- THEN each command SHALL run inside `.venv`

#### Scenario: Celery package is available without a live broker

- GIVEN `uv sync --frozen` succeeded
- WHEN the developer runs `uv run celery --help` or `uv run python -c "import celery"`
- THEN Celery SHALL be importable and its CLI SHALL print usage without requiring a broker or backend

### Requirement: Legacy Requirements Files Removed

The system SHALL NOT keep `requirements.txt` or `requirements-dev.txt` in the repository after PR-3. All dependency declarations SHALL live in `pyproject.toml`, with the resolved tree locked in `uv.lock`.

#### Scenario: Requirements files are absent

- GIVEN a fresh clone after PR-3
- WHEN `git ls-files` runs
- THEN `requirements.txt` and `requirements-dev.txt` SHALL NOT be listed

#### Scenario: Pyproject preserves runtime and dev pins

- GIVEN `pyproject.toml`
- WHEN `[project.dependencies]` and `[project.optional-dependencies.dev]` are read
- THEN all runtime and dev pins formerly declared in `requirements.txt` and `requirements-dev.txt` SHALL be present

### Requirement: Continuous Integration Parity

CI SHALL run only the canonical uv job. The workflow SHALL use `astral-sh/setup-uv@v4`, `uv lock --check`, `uv sync --frozen --extra dev`, and `uv run pytest`. No legacy pip install steps SHALL remain.

#### Scenario: uv job passes

- GIVEN a pull request is opened
- WHEN the CI workflow executes
- THEN the uv job SHALL succeed, validating lockfile freshness and running the test suite with `uv run pytest`

### Requirement: Docker and Compose Compatibility

The system SHALL NOT regress `docker-compose.yml`. If no `Dockerfile` exists, the change SHALL be a no-op and SHOULD document future-image guidance using `uv sync --frozen` with BuildKit cache mounts.

#### Scenario: Compose and Dockerfile are safe

- GIVEN `docker-compose.yml` defines Postgres and Redis and no application `Dockerfile` exists
- WHEN the migration is applied and `docker compose up -d` runs
- THEN services SHALL start without referencing uv files and no build SHALL break

### Requirement: Python Version Contract

The project SHALL declare a single target Python version. `.python-version` SHALL be `3.12` and `pyproject.toml` SHALL require `requires-python = ">=3.12"`. uv SHALL surface a clear error when the active interpreter cannot satisfy that contract.

#### Scenario: Version contract is declared

- GIVEN a fresh clone
- WHEN `.python-version` and `pyproject.toml` are read
- THEN `.python-version` SHALL be `3.12` and `project.requires-python` SHALL be `">=3.12"`

#### Scenario: Mismatch is reported

- GIVEN the active interpreter cannot satisfy `requires-python` (e.g., older than 3.12)
- WHEN `uv sync` runs
- THEN uv SHALL report that Python `>=3.12` is required

### Requirement: Rollback Path

The system SHALL remain reversible to the pip workflow by reverting PR-3 (restoring `requirements*.txt` from git history) and removing uv artifacts.

#### Scenario: Rollback restores pip

- GIVEN the uv CI job, `pyproject.toml`, `.python-version`, `uv.lock`, and uv README blocks are removed and `requirements*.txt` are restored
- WHEN the legacy pip install steps are followed
- THEN the project SHALL install and run as before PR-1
